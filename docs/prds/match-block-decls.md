# PRD: Same-Name Guarded Declarations from `match` Blocks

## Goal

Allow declaration-level `match` blocks (spec §6.4) to produce multiple declarations with the **same name but different types** under mutually-exclusive guards, and let external references resolve via the union type narrowed by the active guard. This is a documented exception to "one name per scope."

## Background

- Spec §6.4 (lines 1363-1384): `match head_type { Hex => sub head : HexHead { ... }; Socket => sub head : SocketHead { ... } }` desugars to per-arm `sub head : ... where head_type == HeadType.Hex { ... }` declarations.
- Quote (line 1382): "The desugaring produces multiple declarations with the same name (`head`) but different types. This is permitted when the guards are mutually exclusive (which exhaustive `match` guarantees). The declarations are treated as a single logical entity -- external references use the shared name (`self.head`), and the type at any reference site is the union of the possible types, narrowed by the active guard. This exception to the normal 'one name per scope' rule applies only to declarations with provably mutually exclusive guards."
- Multi-variant arms (`Socket | Button => sub head : RecessedHead`) collapse to a single arm's worth of decls — the recipe per arm is identical.
- Reference safety (spec §6.3 / §8.10): a reference to `head` from outside the `match` is valid only when the referencing context's guard implies that *some* arm is active. Inside an arm-guarded context, narrowing applies.

## Scope

- Recognize the `match`-block desugaring as a *cluster* of same-name decls.
- Build a logical "group" entity in the symbol table keyed by `(scope, name)` rather than failing the duplicate-name check.
- Type of `self.head` (or `parent.head`) at any reference site = union over the active arms' types, narrowed by the static implication check on the reference site's guard.
- Member access through the group: `head.x` is valid only if `x` is present (with compatible type) in **every** arm of the union active at the reference site; otherwise diagnose with a list of arms missing `x`.
- Reference safety from outside the match: the caller's guard must imply that the match's discriminant lies within the union of arm-discriminants for the same name (in practice: must reach the match block at all).
- Exhaustiveness: piggyback on existing `match`-expression exhaustiveness checking. Non-exhaustive declaration-level `match` blocks are a hard error today and remain so.
- Same-name across `match`-arm and **outside** the match (e.g. `sub head : DefaultHead` outside plus `match head_type { ... => sub head : ... }` inside) is **not** part of this feature. That is a shadow / duplicate (depending on guard relationship) and should diagnose.

## Out of scope

- Same-name decls from non-`match` mutually-exclusive guards (e.g. two manual `where` clauses with negation pairs). v0.1 supports the exception only when generated by exhaustive `match`. Manual mutually-exclusive guards may diagnose as duplicate or as shadowing for now.
- Pattern-matching with payload-bound names — payload-binding grammar + binding semantics are provided by [`v0_6/data-carrying-enums.md`](v0_6/data-carrying-enums.md) (landed; see that PRD §6 G4). Decl-level (§6.4) `match` arms may bind variant payloads once both features land: the payload-binding grammar + semantics are owned by data-carrying-enums, while the same-name-group machinery that wires those binders into the decl-level cluster is owned by this PRD. (This line previously read "v0.1 enums are C-style per §3.8" — superseded now that data-carrying enums have landed.)
- Cross-arm type unification for differently-named decls.

## Acceptance criteria

1. `match head_type { Hex => sub head : HexHead { ... }; Socket => sub head : SocketHead { ... } }` parses and elaborates with no duplicate-name error.
2. Field access `bolt.head.across_flats` where `across_flats` is in **all** arm types resolves; access of an arm-specific field requires either a `where head_type == HeadType.Hex` guard at the reference site or a `match` expression on `head_type` to narrow.
3. Outside the bolt, `bolt.head` typechecks as the arms' union; member access narrows per (2).
4. Non-exhaustive `match` block at decl level — error (existing behavior preserved).
5. Variant pipe `Socket | Button => sub head : RecessedHead { ... }` — one decl, two guard alternatives (`head_type == Socket or head_type == Button`).
6. Same-name decl outside the match — duplicate / shadow diagnostic (no silent merge).
7. Reference safety: accessing `bolt.head` from a `where` block whose condition contradicts every arm's guard is rejected.
8. Tests cover: union typing, narrowing under arm guards, narrowing under outer match, member-access on common-vs-arm-specific fields, exhaustiveness, variant-pipe arms, outside-match collision diagnostic.

## Task breakdown

1. Symbol-table change: introduce a `GuardedDeclGroup { name, arms: Vec<(GuardExpr, Decl)> }` representation. Suppress duplicate-name error when the cluster is recognized as a `match`-arm group.
2. Type-resolution change: when resolving a `MemberAccess` whose receiver resolves to a `GuardedDeclGroup`, build the union of arm-types and apply guard-narrowing using the reference-site guard implication checker (already present for §8.10).
3. Field-access diagnostics: when narrowed type is a union and the requested field is missing in some active arm, list the offending arms.
4. Exhaustiveness gate: ensure the cluster only forms when arms exhaustively cover the discriminant; otherwise per-arm decl flow stays separate (and the existing exhaustiveness error fires).
5. Tests per acceptance criteria.
