# examples/imported_field/openvdb_stress.ri

Demonstrates the `source = imported { format = OpenVDB }` field source kind
(PRD `docs/prds/v0_3/multi-kernel-phase-3.md` §8 Phase 4 task θ).

## What the example models

`fea_stress` is a scalar volumetric field backed by an OpenVDB FloatGrid.  The
file `fixtures/sample.vdb` stores a **cube signed-distance function (SDF)**:

- Grid name: `"vonMises"`
- Half-extent: 1.0 (unit cube)
- Voxel size: 0.1
- Half-width (narrow band): 3.0 voxels → band ≈ ±0.3

`StressProbe` samples the field at three near-surface coordinates on the cube's
+X face (surface at x = 1.0):

| Cell        | x      | Expected sign  |
|-------------|--------|----------------|
| `s_inside`  | 0.85   | SDF < 0 (interior) |
| `s_surface` | 1.00   | SDF ≈ 0 (surface) |
| `s_outside` | 1.15   | SDF > 0 (exterior) |

In a real FEA workflow you would replace `fixtures/sample.vdb` with an actual
von-Mises stress field exported from your solver.

## Why the fixture is not checked in

The repo has zero committed binary fixtures (convention: generate at runtime).
`fixtures/sample.vdb` is generated by the cfg(has_openvdb) smoke test at
`crates/reify-cli/tests/cli_imported_field_eval.rs`, which writes it into a
tempdir automatically when you run:

```
cargo test -p reify-cli --test cli_imported_field_eval -- openvdb_stress_cli_eval_prints_sampled_values
```

## Generating the fixture manually

If you want to generate `fixtures/sample.vdb` by hand for interactive use,
the canonical parameters are:

```
half_extent = 1.0    # cube half-extent in all three axes
voxel_size  = 0.1
half_width  = 3.0    # narrow-band width in voxels (band ≈ ±0.3)
grid_name   = "vonMises"
```

The corresponding Rust code (from the smoke test) is:

```rust
let mut kernel = reify_kernel_openvdb::OpenVdbKernel::new();
let handle = kernel
    .realize_voxel_from_mesh(&verts, &tris, 0.1_f64, 3.0_f64)
    .expect("realize_voxel_from_mesh");
kernel
    .write_vdb_grid(handle, Path::new("fixtures/sample.vdb"), "vonMises")
    .expect("write_vdb_grid");
```

## Running the example

After generating the fixture, evaluate the example with the current directory
set to `examples/imported_field/` so the relative path `fixtures/sample.vdb`
resolves correctly:

```bash
cd examples/imported_field
reify eval openvdb_stress.ri
```

Or from the workspace root using an absolute path with `--cwd` (the CLI resolves
imported field paths relative to the process CWD, not the .ri file's directory):

```bash
REIFY_CWD=examples/imported_field reify eval examples/imported_field/openvdb_stress.ri
```

## Graceful degradation

Without the fixture, `reify eval` emits a `FieldImportFailed` runtime diagnostic
and the probe cells resolve to `Undef`.  The file still **compiles clean** —
the imported block does not read the file at compile time.
